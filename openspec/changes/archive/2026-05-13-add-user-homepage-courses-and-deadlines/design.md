## Context

Сейчас маршрут `src/routes/_authenticated/index.tsx` показывает только набор mock-карточек курсов без общего контекста страницы. Календарь дедлайнов уже реализован как самостоятельный компонент `DeadlinesCalendar`, но вынесен на отдельный маршрут `/calendar`. Требуется превратить главную страницу авторизованного пользователя в dashboard, где оба сценария — обзор курсов и контроль дедлайнов — доступны сразу после входа.

Ограничения текущей реализации:
- проект уже использует TanStack Router, React Query и Tailwind CSS;
- карточка курса и календарь дедлайнов уже имеют собственные UI-компоненты и mock-данные;
- изменение должно быть локальным для фронтенда и не требовать новых внешних зависимостей.

## Goals / Non-Goals

**Goals:**
- Сделать маршрут `/_authenticated/` главной страницей пользователя с двумя основными секциями: курсы и календарь дедлайнов.
- Переиспользовать существующие `CourseCard` и `DeadlinesCalendar`, а не создавать новые независимые виджеты с нуля.
- Сделать layout адаптивным: на мобильных экранах секции идут колонкой, на широких экранах читаются как dashboard.
- Задать предсказуемое отображение при пустом списке курсов и пустом календаре дедлайнов.

**Non-Goals:**
- Подключение реального backend API для курсов или дедлайнов.
- Полный редизайн `CourseCard` или внутренней логики `DeadlinesCalendar`.
- Изменение маршрутизации отдельных страниц курсов или календаря за пределами главной страницы.

## Decisions

### 1. Использовать маршрут `src/routes/_authenticated/index.tsx` как dashboard по умолчанию
Главная страница после авторизации уже привязана к этому маршруту, поэтому логично расширить его до пользовательского overview. Это уменьшает количество переходов и делает новую страницу естественной точкой входа.

**Alternatives considered:**
- Создать отдельный маршрут `/dashboard` и редиректить на него после логина. Отклонено, потому что это добавляет лишнюю навигацию и не дает заметных преимуществ для текущего объема функциональности.

### 2. Собрать страницу из двух независимых секций: списка курсов и блока календаря
Секция курсов должна рендерить список карточек через `CourseCard`, а секция дедлайнов — существующий `DeadlinesCalendar`. Для маршрута стоит добавить структурные обертки, заголовки секций и ограничения по размерам, не меняя публичные API существующих компонентов без необходимости.

**Alternatives considered:**
- Встраивать календарь внутрь секции курсов или смешивать курсы и дедлайны в одном списке. Отклонено, потому что это ухудшает читаемость и усложняет адаптивность.

### 3. Вынести данные о курсах из route-компонента в отдельный источник данных уровня feature
Текущий route хранит примитивный массив цветов и дублирует demo-данные прямо в JSX. Для dashboard лучше иметь отдельный mock-источник и/или query hook по аналогии с `deadlines-calendar`, чтобы route отвечал за композицию страницы, а не за подготовку данных.

**Alternatives considered:**
- Оставить все mock-данные inline в маршруте. Отклонено, потому что это мешает повторному использованию и усложняет дальнейшую замену mock на API.

### 4. Использовать адаптивный dashboard layout с приоритетом курсов на узких экранах
На мобильных экранах секции должны идти вертикально: сначала курсы, затем календарь. На широких экранах допустим layout с двумя колонками, где календарь занимает отдельный крупный блок, а курсы остаются в читаемой сетке. Высота календаря должна быть ограничена контейнером страницы, чтобы он оставался функциональным без захвата всей страницы.

**Alternatives considered:**
- Показывать только сокращенный календарь на главной странице и оставлять полный календарь на отдельном маршруте. Отклонено, потому что пользовательский запрос прямо требует календарь дедлайнов на главной странице.

### 5. Пустые состояния должны объяснять отсутствие данных, а не оставлять пустые контейнеры
Если у пользователя нет курсов, секция курсов должна показывать понятное сообщение. Если по текущему диапазону нет дедлайнов, календарь должен оставаться доступным, а отсутствие событий должно объясняться через существующее поведение календаря или явное пустое состояние секции.

**Alternatives considered:**
- Скрывать пустые секции полностью. Отклонено, потому что это делает интерфейс нестабильным и ухудшает предсказуемость страницы.

## Risks / Trade-offs

- [Route станет более сложным compositional-компонентом] → Вынести получение данных и секции в отдельные feature-level компоненты или hooks, оставив route тонким.
- [Mock-данные курсов и дедлайнов могут выглядеть несогласованно] → Использовать единые demo-данные по цветам, названиям и структурам, чтобы страница выглядела целостно до интеграции с API.
- [Календарь — тяжелый визуальный блок для главной страницы] → Ограничить его высоту и встроить в отдельную секцию с предсказуемым scroll-контейнером.
- [Отдельный маршрут `/calendar` может дублировать часть нового сценария] → Сохранить маршрут как специализированный экран, а главную страницу сделать обзорной точкой входа.

## Migration Plan

1. Обновить композицию маршрута `/_authenticated/` и добавить page-level layout.
2. Вынести mock-данные/получение курсов из route в отдельный модуль.
3. Подключить `DeadlinesCalendar` к новой секции dashboard с подходящими размерами.
4. Проверить отображение на мобильных и desktop-размерах.

Rollback: вернуть прежний JSX маршрута `src/routes/_authenticated/index.tsx`; изменение не требует миграции данных.
